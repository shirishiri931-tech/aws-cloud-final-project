# 3. ארכיטקטורה

## 3.1 שירותי AWS

המערכת בנויה כ‑**Serverless** מלא באזור `us-east-1`.

| שירות | תפקיד |
|--------|--------|
| **Amazon S3** | אחסון פרטי לגרסאות הקבצים. כל גישה דרך Presigned URLs בלבד. |
| **Amazon DynamoDB** | `DocumentFlow-Documents` (מטא‑דאטה) + `DocumentFlow-VersionHistory` (היסטוריית גרסאות). |
| **AWS Lambda** | 8 פונקציות לוגיקה עסקית (Python 3.11). |
| **Amazon API Gateway** | REST API עם נתיבים תומכי CORS. |
| **Amazon Cognito** | אימות משתמשים ותפקידים מבוססי קבוצות (Admin/PMO/Reviewer). |
| **Amazon SES** | התראות דוא"ל לאורך מחזור החיים של המסמך. |
| **Amazon EventBridge** | משימה יומית מתוזמנת לסימון מסמכים באיחור. |
| **Amazon CloudWatch** | לוגים וניטור לכל הפונקציות. |
| **AWS IAM** | תפקיד הרצה עם הרשאות מינימום (least‑privilege). |

## 3.2 זרימת נתונים (Data Flow)

הדיאגרמה מציגה את זרימת הנתונים בין רכיבי המערכת: המשתמשים (PMO, אחראים,
מבקרים) ניגשים ל‑Frontend, האימות מתבצע מול Cognito, הבקשות עוברות דרך
API Gateway אל פונקציות ה‑Lambda הניגשות ל‑DynamoDB ו‑S3, ומשם נשלחות התראות
דוא"ל (SES) ונרשמים לוגים (CloudWatch). EventBridge מפעיל את `overdue_check`
מדי יום.

## 3.3 מודל סטטוסים (Status Workflow)

```
Not Started → In Progress → Ready for Review → Under Review
   → Approved | Revision Required | Rejected
```

המעברים מונעים בחלקם על‑ידי העלאות: `upload_version` מקדם את הסטטוס לפי
`versionType` (`Submitted`/`Revised` → **Ready for Review**, `Reviewed` →
**Under Review**), כך שהתקדמות מחזור החיים אינה תלויה רק בעדכון ידני.
`Overdue` מחושב אוטומטית (`overdue_check`) ואינו סטטוס שנבחר ידנית.

## 3.4 מודל הרשאות

- **סינון מסמכים בצד הלקוח:** Admin/PMO רואים את כל המסמכים; משתמש רגיל רואה רק
  מסמכים שבהם הוא `assignedUserEmail` או `reviewerEmail`.
- **תפקידים מ‑Cognito Groups:** התפקיד הראשי נגזר מ‑`cognito:groups` שב‑ID Token.
- **least‑privilege:** תפקיד ה‑Lambda משתמש ב‑policy ייעודי ל‑SES
  (`ses:SendEmail`, `ses:SendRawEmail`) ולא ב‑`AmazonSESFullAccess`.

## 3.5 הערות אבטחה

- דלי ה‑S3 פרטי לחלוטין; כל גישה דרך Presigned URLs בעלי תוקף מוגבל (העלאה: שעה, הורדה: 300 שניות).
- אין סיסמאות קשיחות בקוד; Cognito מייצר סיסמה זמנית אקראית בעת יצירת משתמש.
- SES נמצא כעת במצב Sandbox — התראות נשלחות רק לכתובות מאומתות.
- שגיאות `500` מחזירות הודעה גנרית (`Internal server error`) ואינן מדליפות פרטי
  חריגה ללקוח; הקלט מאומת בצד השרת (פורמט מועד, כתובות דוא"ל, שם קובץ).

## 3.6 מבנה ה‑Frontend

לאחר השדרוג, אפליקציית ה‑React מאורגנת לפי אחריות (במקום קובץ `App.jsx` יחיד):

| תיקייה | תפקיד |
|--------|--------|
| `components/ui` | רכיבי בסיס לשימוש חוזר (Button, Modal, Table, Toast, Pagination, FilterTabs…). |
| `components/auth` · `admin` · `documents` | מסכים לפי תחום (התחברות, ניהול משתמשים, מסמכים). |
| `context` | `AuthContext` ו‑`ToastContext` לניהול מצב גלובלי. |
| `hooks` | `useDocuments`, `useUsers`, `useHistory`, `useAuth`, `useToast` — לוגיקת נתונים. |
| `lib` | `api.js` (לקוח REST), `cognito.js`, `roles.js`, `statusConfig.js`, `versionConfig.js`. |

הפרדה זו ממקמת את קריאות ה‑API ב‑`lib/api.js` ואת לוגיקת המצב ב‑hooks, כך
שהמסכים נשארים דקים — ללא שינוי בחוזה ה‑REST מול ה‑Backend.

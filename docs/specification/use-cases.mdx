# 2. Use Cases (תרחישי שימוש)

להלן שבעה תרחישי שימוש (מעבר לדרישת המינימום של חמישה), כל אחד נגזר ישירות
מזרימה אמיתית בקוד — אפליקציית ה‑React (`frontend/src/App.jsx`) ופונקציות
ה‑Lambda (`backend/functions/`).

| # | תרחיש | שחקן ראשי | Lambda / רכיב |
|---|--------|-----------|----------------|
| UC‑1 | יצירת משתמש חדש | Admin | `manage_users` |
| UC‑2 | התחברות ראשונה והחלפת סיסמה | כל משתמש | Cognito |
| UC‑3 | יצירת מסמך והקצאת אחראי | PMO | `create_document` |
| UC‑4 | העלאת גרסה להגשה | אחראי (Assigned) | `upload_version` |
| UC‑5 | סקירה ושינוי סטטוס | Reviewer | `update_status` |
| UC‑6 | מחזור תיקון (Revision) | אחראי + מבקר | `upload_version` + `update_status` |
| UC‑7 | סימון Overdue אוטומטי | מערכת (מתוזמן) | `overdue_check` |

---

## UC‑1 — יצירת משתמש חדש (Admin)

- **שחקן ראשי:** Admin
- **תנאי קדם:** ה‑Admin מחובר ונמצא בלוח הניהול (Admin Panel).
- **טריגר:** לחיצה על **"+ New User"**.

**זרימה ראשית:**
1. ה‑Admin ממלא שם פרטי, שם משפחה, דוא"ל ובוחר תפקיד (`Admin` / `PMO` / `Reviewer`).
2. הבקשה נשלחת ל‑`POST /users` → פונקציית `manage_users`.
3. המערכת מייצרת **סיסמה זמנית אקראית** (`generate_temp_password`) ויוצרת משתמש
   ב‑Cognito עם `email_verified=true`.
4. המשתמש משויך לקבוצה (Group) המתאימה לתפקידו.
5. Cognito שולח דוא"ל הזמנה עם הסיסמה הזמנית.

- **תוצאה (Postcondition):** משתמש חדש קיים במצב `FORCE_CHANGE_PASSWORD` ומופיע בטבלת המשתמשים.
- **חריגים:** דוא"ל חסר → `400 Email is required`; תפקיד לא חוקי → `400 Invalid role`.

---

## UC‑2 — התחברות ראשונה והחלפת סיסמה

- **שחקן ראשי:** כל משתמש חדש.
- **תנאי קדם:** המשתמש קיבל דוא"ל הזמנה עם סיסמה זמנית (UC‑1).

**זרימה ראשית:**
1. המשתמש מזין דוא"ל וסיסמה זמנית במסך ההתחברות.
2. Cognito מחזיר `newPasswordRequired` והאפליקציה מציגה טופס סיסמה חדשה.
3. המשתמש קובע סיסמה קבועה (`completeNewPasswordChallenge`).
4. בהתחברות מוצלחת נשלף `cognito:groups` מתוך ה‑ID Token, ונקבע התפקיד הראשי.
5. ניתוב לפי תפקיד: **Admin** → לוח ניהול; **PMO/Reviewer** → לוח המסמכים.

- **תוצאה:** מצב המשתמש הופך ל‑`CONFIRMED` ("Active") ונפתח session תקף.
- **חריגים:** סיסמה שגויה → הודעת שגיאה מ‑Cognito; session שפג → חזרה למסך התחברות.

---

## UC‑3 — יצירת מסמך והקצאת אחראי (PMO)

- **שחקן ראשי:** PMO (או Admin).
- **תנאי קדם:** ה‑PMO מחובר; קיימים משתמשים שניתן להקצות.
- **טריגר:** לחיצה על **"+ New Document"**.

**זרימה ראשית:**
1. ה‑PMO ממלא כותרת, שם פרויקט, קטגוריה ומועד יעד.
2. בוחר **אחראי (Responsible User)** ו**מבקר (Reviewer)** מתוך רשימת המשתמשים
   (נטענת מ‑`GET /users`).
3. הבקשה נשלחת ל‑`POST /documents` → פונקציית `create_document`.
4. נוצרת רשומה ב‑`DocumentFlow-Documents` עם `status = "Not Started"`,
   `currentVersion = 0` ו‑`documentId` (UUID).
5. נשלח דוא"ל לאחראי: **"New Task Assigned"** (Amazon SES).

- **תוצאה:** מסמך חדש מופיע בטבלה; האחראי קיבל התראה.
- **חריגים:** שדה חובה חסר (`title`, `projectName`, `category`, `assignedUserId`, `deadline`) → `400 Missing required field`;
  פורמט מועד לא תקין (שאינו `YYYY-MM-DD`) → `400 Invalid deadline format`;
  כתובת דוא"ל לא תקינה (`assignedUserEmail`/`reviewerEmail`) → `400 Invalid email format`.

## UC‑4 — העלאת גרסה להגשה (אחראי)

- **שחקן ראשי:** המשתמש האחראי (Assigned User).
- **תנאי קדם:** קיים מסמך שהמשתמש מוקצה אליו.
- **טריגר:** לחיצה על **"Upload Submitted"** ובחירת קובץ.

**זרימה ראשית:**
1. האפליקציה קוראת ל‑`POST /documents/{id}/versions` → `upload_version`.
2. הפונקציה מחשבת `new_version = currentVersion + 1` ומייצרת **Presigned PUT URL**
   ל‑S3 (תוקף שעה), עם מפתח `documents/{id}/versions/v{n}/{fileName}`.
3. הדפדפן מעלה את הקובץ **ישירות ל‑S3** באמצעות ה‑URL החתום (`PUT`).
4. רשומת המסמך מתעדכנת (`currentVersion`, `s3Key`, `currentFileName`), והסטטוס
   נגזר מ‑`versionType`: `Submitted` → **Ready for Review**, `Reviewed` →
   **Under Review**, `Revised` → **Ready for Review**, אחרת **In Progress**.
   במקביל נוספת רשומה ל‑`DocumentFlow-VersionHistory`.
5. אם `versionType = "Submitted"` ויש מבקר — נשלח דוא"ל למבקר: **"Document Ready for Review"**.

- **תוצאה:** גרסה חדשה נשמרת ב‑S3 ובהיסטוריה; הסטטוס מתקדם אוטומטית והמבקר קיבל התראה.
- **חריגים:** `id` חסר או `fileName` ריק → `400`; מסמך לא קיים → `404 Document not found`.

---

## UC‑5 — סקירה ושינוי סטטוס (Reviewer)

- **שחקן ראשי:** Reviewer.
- **תנאי קדם:** קיים מסמך עם גרסה מוגשת שהמבקר מוקצה אליו.

**זרימה ראשית:**
1. המבקר פותח את מסך פרטי המסמך ולוחץ **Download** — נוצר **Presigned GET URL**
   (`get_download_url`, תוקף 300 שניות) להורדה מאובטחת.
2. לאחר הסקירה, המבקר יכול להעלות גרסה מוערת (**"Upload Reviewed"**, `versionType=Reviewed`).
3. המבקר פותח את חלון **Change Status** ובוחר סטטוס:
   - **Approved** → `PUT /documents/{id}/status` → דוא"ל ל‑PMO היוצר ("Document Approved").
   - **Revision Required** / **Rejected** → דוא"ל לאחראי ("Action Required").
4. `update_status` מאמת שהסטטוס נמצא ב‑`VALID_STATUSES`, מעדכן את הרשומה ושומר הערה.

- **תוצאה:** סטטוס המסמך מתעדכן; הגורם הרלוונטי קיבל התראה.
- **חריגים:** סטטוס לא חוקי → `400 Invalid status`; `id`/`status` חסר → `400`.

---

## UC‑6 — מחזור תיקון (Revision Cycle)

- **שחקנים:** אחראי + מבקר.
- **תנאי קדם:** מסמך בסטטוס **Revision Required** (מ‑UC‑5).

**זרימה ראשית:**
1. האחראי מקבל דוא"ל "Action Required" ופותח את המסמך.
2. מעלה גרסה מתוקנת באמצעות **"Upload Revised"** (`versionType=Revised`) — גרסה חדשה נוספת להיסטוריה.
3. האחראי/מערכת מסמנים שהמסמך מוכן לסקירה חוזרת; המבקר מקבל התראה.
4. המבקר חוזר על UC‑5 עד אישור (**Approved**) או דחייה (**Rejected**).

- **תוצאה:** כל איטרציה נשמרת כגרסה נפרדת בהיסטוריה; הגרסה הנוכחית מסומנת תמיד.
- **הערה:** המחזור יכול לחזור על עצמו מספר פעמים — אין הגבלת מספר גרסאות.

---

## UC‑7 — סימון Overdue אוטומטי (מערכת)

- **שחקן ראשי:** המערכת (משימה מתוזמנת).
- **טריגר:** טריגר יומי של **Amazon EventBridge**.

**זרימה ראשית:**
1. EventBridge מפעיל את `overdue_check` פעם ביום.
2. הפונקציה סורקת מסמכים שבהם `deadline < today` והסטטוס אינו סופי
   (`Approved` / `Completed` / `Overdue`).
3. לכל מסמך תואם מעודכן הסטטוס ל‑**Overdue** ומתעדכן `lastUpdated`.

- **תוצאה:** מסמכים שחרגו ממועד היעד מסומנים אוטומטית כ‑Overdue ובולטים ב‑UI.
- **הערה:** Overdue הוא **ערך מחושב** ואינו ניתן לבחירה ידנית מתוך תפריט הסטטוס.
